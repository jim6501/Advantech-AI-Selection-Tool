# RAG Chatbot 實作指南
選型工具 · Chatbot 架構設計、實作順序、注意事項與驗證方法

---

## 1. 整體架構

本工具 Chatbot 採 Hybrid Search 架構，根據問題類型走不同的查詢路徑，確保規格數值準確、語意回答豐富。

| 問題類型 | 查詢路徑 | 資料來源 | 範例 |
|----------|----------|----------|------|
| 規格查詢 | 結構化查詢 | MongoDB 規格資料庫 | 這款支援幾個 Port？工作溫度範圍？ |
| 場景 / 推薦 | RAG 語意檢索 | DS、Sales Kit、Case Study | 適合工廠自動化的產品有哪些？ |
| 混合類問題 | 結構化篩選 + RAG | MongoDB + 知識庫 | 支援 PROFINET、適合軌道交通的型號 |
| 文件解析 | AI 萃取 + 結構化查詢 | 使用者上傳文件 | 上傳 QFS 或客戶信件後自動選型 |

> **架構重點**  
> 兩條查詢路徑共用同一份 MongoDB 規格資料庫，選型引擎和 Chatbot 使用相同資料來源，不需維護兩套資料。Google Sheet 為 PM 維護規格的來源端，透過同步機制更新至 MongoDB。問題分類層 Phase 1 先用關鍵字規則，穩定後再升級為 LLM 分類。

---

## 2. 實作順序

### Phase 1 — 同步機制建立 + 基礎查詢

**目標**
- Google Sheet 規格資料穩定同步至 MongoDB
- 建立 Chatbot 規格問答的基礎能力
- 驗證資料同步正確性與查詢準確率

**實作項目**
- 建立排程同步（Polling）：每天定時自動比對 Google Sheet 與 MongoDB，有差異則更新，同步結果寫入 log
- 建立手動觸發同步：後台管理介面加入「立即同步」按鈕，PM 有重要更新（新產品、EOL）時可立即執行，同步後顯示新增 / 更新 / 錯誤筆數
- 建立同步驗證腳本：每次同步後自動執行筆數比對、必填欄位檢查、資料型態驗證，有錯誤發通知給 PM
- 複用現有前端查詢 API（`/submitProdType`），確認支援多條件組合、範圍查詢、模糊比對
- 建立問題分類層（Hybrid Router）：文件優先判定 → Gemini Flash 意圖分類 → 信心值 Fallback
- 串接 LLM 生成自然語言回答（規格數值由 MongoDB 提供，LLM 只負責組織語言）

**注意事項**
- MongoDB 已有規格資料，同步前先確認現有欄位結構與 Google Sheet 欄位是否完全對應，有落差需先補齊
- 範圍值欄位（如溫度 -40°C ~ 75°C）確認已拆成 min / max 兩欄儲存，才能支援範圍查詢
- LLM 回答規格類問題時，數值必須來自 MongoDB，不能讓 LLM 自己生成數字
- 排程同步建議設在離峰時段（如凌晨），避免同步期間影響前端查詢效能

---

### Phase 2 — 知識庫建置 + 語意檢索 + 混合查詢

**目標**
- Chatbot 能回答場景、推薦理由、案例參考類問題
- 含規格條件與情境描述的混合問題可正確處理
- 知識庫更新後自動重新索引

**實作項目**
- 文件前處理：PDF 萃取文字（建議使用 LlamaParse，處理表格和圖片）
- 文件分塊策略：DS 按規格表段落切分，Sales Kit 按產品賣點切分，Case Study 按案例切分
- 向量化並存入向量資料庫（建議 Qdrant 或 Weaviate，可自架），每份文件加上型號 metadata
- 建立 RAG 檢索 API，支援依型號清單過濾搜尋範圍（用於混合查詢第二段）
- 實作混合查詢流程：第一段呼叫 `/submitProdType` 取得候選清單 → 第二段對候選型號做語意檢索 → 交給 Gemini Pro 整合生成回答
- 處理候選清單為空的邊界情況：直接回答找不到符合規格的型號，不進行第二段語意搜尋
- 建立文件更新觸發機制：DS 有新版本時自動重新索引對應型號

**注意事項**
- DS 的規格表用純文字萃取效果差，需要特別處理表格結構，建議先用 10 份 DS 測試萃取品質再全量處理
- 文件分塊大小影響檢索精準度，建議 chunk size 300~500 token，並保留前後文 overlap
- 不同來源的文件（DS vs Sales Kit vs Case Study）要加 metadata 標注，回答時才能標注引用來源
- 混合查詢的語意搜尋必須限定在候選型號範圍內，不可搜尋全庫，否則會混入規格不符的型號
- Sales Kit 和 Case Study 如果是 PowerPoint 格式，需要額外的前處理工具

#### 混合查詢流程細節

| 步驟 | 執行內容 | 說明 |
|------|----------|------|
| 1. 輸入解析 | Gemini Flash 拆解使用者輸入 | 分離出結構化條件（Port、協定、溫度）與情境描述（軌道交通、戶外、震動耐受） |
| 2. 結構化查詢 | 呼叫現有 `/submitProdType` API | 取得符合規格條件的候選型號清單，複用前端已有的查詢 API |
| 3. 空值檢查 | 判斷候選清單是否為空 | 候選清單為空時直接回答「無符合規格型號」，不進行第四步，避免搜出規格不符的結果 |
| 4. 語意檢索 | RAG 只搜尋候選型號的知識庫 | 以情境描述為查詢，在候選型號對應的 DS / Case Study 內容中做語意比對 |
| 5. 整合生成 | Gemini Pro 合併兩段結果 | 說明推薦哪款、規格符合哪些條件、情境上為什麼適合，並標注引用來源 |

---

### Phase 3 — 文件解析選型（內部功能）

**目標**
- 內部人員可上傳 QFS 或客戶信件，由 Chatbot 自動萃取規格條件並執行選型
- 資訊不足時主動追問，確保選型條件完整

**實作項目**
- 實作文件上傳介面（支援 PDF、Word、常見信件格式）
- 建立 AI 萃取流程：上傳文件 → 萃取規格條件 → 結構化輸出
- 設計追問邏輯：必填欄位缺少時，Chatbot 主動詢問補充
- 萃取結果帶入 Phase 1 的結構化查詢，回傳符合型號
- 實作多輪對話記憶（session 管理），同一案件內保留上下文
- 選型結果可儲存為案件紀錄，綁定當時 Spec 版本

**注意事項**
- QFS 格式可能因客戶不同而差異很大，萃取邏輯要有容錯設計，萃取失敗時要提示使用者手動確認
- 多輪對話的 context 長度有限制，長對話要實作 context 壓縮或摘要機制
- 上傳文件的資安處理：文件不應永久儲存在伺服器，處理完後要清除

## 3. 跨階段注意事項

### 資料品質與同步

- MongoDB 已有規格資料，每次同步後自動執行驗證腳本：筆數比對、必填欄位檢查、資料型態驗證，有異常立即通知 PM
- 排程同步（每日定時）+ 手動觸發同步（後台按鈕）並行，確保重要更新可即時反映
- DS 文件版本要與 MongoDB 規格同步，知識庫文件版本需對應資料庫版本，避免 Chatbot 引用舊版規格

---

### 意圖分類：Hybrid Router

Chatbot 收到使用者輸入後，需先判斷問題類型再走對應的查詢路徑。建議採用三層判斷機制：

| 層級 | 判斷方式 | 說明 |
|------|----------|------|
| 第一層：文件優先判定 | 規則偵測 | 偵測是否有上傳檔案，或輸入包含「這份文件」、「如附件」、「以下是客戶需求」等關鍵字，若是則直接導向文件解析流程 |
| 第二層：LLM 意圖分類 | Gemini Flash | 將使用者輸入丟給輕量模型，回傳 JSON 格式分類結果 |
| 第三層：信心值 Fallback | 閾值判斷 | 信心值低於 0.7 或 intent 為 unclear 時，預設走混合查詢，或反問使用者澄清意圖 |

**LLM 分類回傳格式**
```json
{
  "intent": "spec | scene | hybrid | unclear",
  "confidence": 0.85
}
```

> **Hybrid Router 注意事項**  
> 反問使用者只用在真的無法判斷的情況，避免頻率過高讓使用者覺得 Chatbot 能力不足。LLM 分類 Prompt 建議加入 few-shot 範例（如「Port 數查詢 → spec、適合軌道交通 → scene」），工業專業詞彙的分類準確率會顯著提升。分類結果建議在回答末尾標注資料來源（規格資料庫 / DS 知識庫），方便 PM 追蹤和除錯。

---

### Multi-Model 架構：不同任務使用不同模型

不同任務對模型的需求不同，建議依任務特性分配模型，以平衡品質與成本。本工具以 Gemini 系列為主要模型：

| 任務 | 建議模型 | 理由 |
|------|----------|------|
| 意圖分類（Hybrid Router） | Gemini 2.0 Flash | 只需輸出簡單 JSON，速度快、成本低、延遲低 |
| 規格問答 / RAG 語意回答 | Gemini 1.5 Flash | 平衡速度與理解品質，處理中等複雜度的回答 |
| 選型報告生成 | Gemini 1.5 Pro | 長文生成、多來源整合，需要較強的組織與推理能力 |
| 文件解析萃取（QFS / 信件） | Gemini 1.5 Flash | 結構化萃取任務，Flash 已足夠，成本較低 |

> **LLM Gateway 層（重要）**  
> 不同任務呼叫不同模型，後端必須設計統一的 LLM Gateway 層，所有任務都透過這個層呼叫 API，而不是各自直接打 Gemini API。好處是：未來要換模型只需改 Gateway 設定，不用改各任務程式碼；也可以在這一層統一做 retry、錯誤處理、用量監控與費用追蹤。RD 在規劃後端架構時應提前設計此層。

---

### 回答品質控制

- 規格數值類回答必須標注來源型號和欄位，不能讓 LLM 自己推算數值
- 語意類回答要附上引用來源（文件名稱、版本），讓使用者可以自行驗證
- 設計回答信心度機制：檢索結果相似度低於閾值時，回答「找不到相關資料」而非生成不確定的答案

## 4. 驗證方法

驗證分為兩層：**自動化初步驗證**（每次資料更新或程式異動後自動觸發）和**人工驗收**（每個 Phase 結束後由 PM / PAE 執行）。兩層互補，自動化驗證降低人工成本，人工驗收確保品質符合實際使用需求。

---

### 自動化初步驗證

#### 資料同步驗證（每次同步後自動執行）

| 驗證項目 | 檢查內容 | 異常處理 |
|----------|----------|----------|
| 筆數比對 | Google Sheet 筆數 = MongoDB 筆數 | 數量不符時擋住同步，發通知給 PM |
| 必填欄位 | 型號名稱、Port 數、溫度範圍等不得為空 | 列出異常筆數和欄位，不讓髒資料進資料庫 |
| 資料型態 | Port 數為整數、溫度為數值、min ≤ max | 格式錯誤筆數超過閾值則同步失敗 |
| 同步日誌 | 記錄新增 / 更新 / 刪除筆數和時間 | 每次同步結果可在後台管理介面查看 |

#### 查詢層自動化測試（程式碼異動後自動執行）

建立黃金測試集（30～50 題），每題附上正確答案，每次程式碼異動後自動跑完整測試集並比對結果：

| 測試類型 | 範例 | 通過標準 |
|----------|------|----------|
| 單一條件查詢 | Port 數 = 8 的型號 | 結果與 MongoDB 直接查詢一致 |
| 多條件組合 | Port ≥ 8 且支援 PROFINET | 無漏選、無誤選 |
| 範圍條件 | 溫度範圍涵蓋 -40°C | 邊界值判斷正確 |
| 不存在型號 | 查詢不存在的型號 | 回傳空結果，不捏造資料 |

#### RAG 品質自動評估（使用 RAGAS 框架）

準備測試題集（問題 + 標準答案），使用 RAGAS 開源框架自動評分，不需人工逐題檢查：

| 評估指標 | 說明 | 建議閾值 |
|----------|------|----------|
| Faithfulness | 回答有無幻覺，內容是否來自知識庫 | > 0.8 |
| Answer Relevancy | 回答與問題的相關程度 | > 0.75 |
| Context Precision | 檢索到的段落是否精準 | > 0.7 |
| Context Recall | 有無漏掉重要資訊 | > 0.7 |

#### 意圖分類準確率（Hybrid Router 驗證）

準備分類測試集，每題標注正確 intent，自動跑分類準確率：

- 測試集涵蓋 spec、scene、hybrid、unclear 四種類型，各至少 15 題
- 目標分類準確率 > 90%，低於此值需調整 Prompt 或關鍵字規則
- few-shot 範例異動後必須重跑此測試

> **自動化驗證整合建議**  
> 以上四層驗證建議整合成 CI 流程，資料同步或程式碼異動時自動觸發，跑完輸出驗證報告。人工驗收只需確認報告中的分數是否達標，不需逐題手動測試，大幅降低每次更新的驗收成本。

---

### 人工驗收（各 Phase 結束後）

#### Phase 1 驗收：規格查詢 + 同步機制

| 測試項目 | 測試方式 | 通過標準 |
|----------|----------|----------|
| 單一規格查詢 | 詢問特定型號的 Port 數、溫度範圍等 | 數值與 MongoDB 完全一致 |
| 多條件組合查詢 | 同時指定 Port 數 + 協定 + 溫度範圍 | 結果包含所有符合型號，不漏不多 |
| 範圍條件查詢 | 詢問「工作溫度 -40°C 以下」的型號 | 範圍判斷正確，不誤納邊界值 |
| 排程同步驗證 | Google Sheet 修改後等待排程，確認 MongoDB 是否更新 | 同步延遲在排程時間內，數值一致 |
| 手動同步驗證 | 按下後台「立即同步」按鈕後確認結果 | 同步報告顯示正確的新增 / 更新筆數 |
| 不存在型號 | 詢問不存在的型號規格 | 明確回答找不到，不捏造資料 |

#### Phase 2 驗收：語意檢索 + 混合查詢

| 測試項目 | 測試方式 | 通過標準 |
|----------|----------|----------|
| 場景推薦 | 描述應用場景，詢問推薦型號 | 推薦結果與規格條件一致，理由有引用來源 |
| 案例查詢 | 詢問特定產業的應用案例 | 回答來自 Case Study，有標注文件來源 |
| 混合查詢 | 同時含規格條件和場景描述 | 規格條件走 MongoDB、場景說明走 RAG，結果合併正確 |
| 候選清單為空 | 輸入不存在的規格組合 | 直接回答找不到，不進行語意搜尋 |
| 知識庫更新 | DS 更新後重新問相關問題 | 回答反映最新版本內容 |
| 無相關資料 | 詢問知識庫沒有的內容 | 明確告知找不到，不生成不確定答案 |

#### Phase 3 驗收：文件解析選型

| 測試項目 | 測試方式 | 通過標準 |
|----------|----------|----------|
| QFS 解析 | 上傳不同格式的 QFS 文件 | 必填規格欄位萃取正確率 > 90% |
| 信件解析 | 上傳含規格需求的客戶信件 | 模糊描述能正確對應到規格條件 |
| 追問機制 | 上傳資訊不完整的文件 | Chatbot 主動詢問缺少的必填欄位 |
| 多輪對話 | 在同一對話中補充多次資訊 | 前後輪資訊正確累積，不遺忘 |
| 選型結果 | 完整提供條件後執行選型 | 結果與手動篩選一致 |

> **人工驗收建議**  
> 建議 PM 或 PAE 準備 20~30 題測試題集（涵蓋三種問題類型），每次大版本更新後重跑完整測試。特別注意邊界條件（範圍臨界值、不存在的型號、格式異常的文件），這類問題最容易在上線後才被發現。自動化驗證通過後再進行人工驗收，效率最高。
